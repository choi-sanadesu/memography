# Query 출력 템플릿

`/query` 스킬은 세 가지 리포트 모드 중 하나로 답한다: **확정 / 부분 / 모름**.

모든 모드는 quote-heavy — 근거 파일의 실제 본문을 발췌해 답변을 뒷받침한다. 파일 경로는 Obsidian 위키링크 형식(`[[path]]`)으로 노출해 사용자가 바로 이동할 수 있게 한다.

각 답변 끝에 **log.md 기록 완료 한 줄 명시** (retrieval.md Phase 3 준수).

---

## 1. 확정 답변 — 충분한 증거

retrieval.md Phase 0~2 진행 중 충분한 file-back 증거가 나왔고 질의 의도가 해소된 경우.

```
## 답변
{요약 2~5줄. 근거 파일이 실제로 말하는 것만 합성}

## 근거

### [[{path1}]] (Phase 1: entities 직행, hop 0)
> {발췌 3~10줄. 원문 그대로}

### [[{path2}]] (Phase 2: related.entities, hop 1, L2)
> {발췌}

### [[{path3}]] (Phase 2: 본문 [[...]] hop 2, L1)
> {발췌 또는 frontmatter summary}

## 질의 해석
- 키워드: {extracted_keywords}
- Phase 1 매핑: {질문 유형} → {진입점}
- 사용 Phase: {0, 1, 2 중 활성된 것}
- 누적 토큰: {N}/5000

## log
- log.md에 query 이벤트 기록 완료 (참조 {N}건)
```

### 규칙

- `## 답변`은 근거 파일 발췌 범위 안에서만 합성. 발췌에 없는 내용 추가 금지.
- `## 근거`는 최대한 원문 그대로 인용. 길이 조절 시 `...`로 생략 표기.
- 근거 파일은 3~5개로 제한. 그 이상이면 가장 신뢰도 높은 것부터 선별.
- Phase 라벨 + hop + 로드 깊이(L1/L2) 함께 표기 → 사용자가 증거 출처와 비용 파악 가능.

---

## 2. 부분 답변 — 증거가 있지만 불충분

Phase 탐색 결과 관련 파일은 나왔지만 hop 2 한도 내에서 질의 일부만 해소되거나, 토큰 5k 한도로 일부 페이지 미탐색.

```
## 부분 답변
{해소된 부분 요약}

## 근거

### [[{path1}]] (Phase 2: 본문 hop 2, L2)
> {발췌}

## 주의
- {해소되지 않은 부분 명시}
- 한도 도달: {hop 2 종료 | 토큰 5k 초과 | 폴백 _index.md까지 시도}
- 질의를 더 구체화하면 정답률이 오른다: {제안}

## 질의 해석
- 키워드: {extracted_keywords}
- Phase 1 매핑: {질문 유형} → {진입점}
- 사용 Phase: {0, 1, 2}
- 누적 토큰: {N}/5000

## log
- log.md에 query 이벤트 기록 완료 (참조 {N}건, 결과: 부분)
```

### 규칙

- `## 주의` 섹션을 반드시 포함. 사용자가 결과를 얼마나 믿을지 판단할 재료를 준다.
- 한도 도달 사유 명시 (hop 한도 vs 토큰 한도 vs 폴백 소진).
- 상충되는 근거가 있으면 양쪽 모두 인용하고 어느 쪽이 맞는지 단정 금지.

---

## 3. 모름 — 증거 0

Phase 0~2 절차 종료까지 file-back 증거 0:

- Phase 0 hot.md: 매칭 0
- Phase 1 폴더 직행: 매칭 0
- Phase 2 hop 2 + 폴백 `_index.md`·`index.md`: 매칭 0

```
## 모름

볼트에서 "{question}"에 대한 근거를 찾지 못했다.

- Phase 0 (hot.md): 매칭 0
- Phase 1 (폴더 직행 — {진입점}): 매칭 0
- Phase 2 (hop 2 + 폴백): 매칭 0

### 질의 재구성 제안
1. {구체적 제안 1 — 예: 동의어·다른 표기 (`aliases:` 필드)}
2. {구체적 제안 2 — 예: 폴더 한정 변경 또는 제거}
3. {구체적 제안 3 — 예: 엔티티명을 더 넓은 범주로}

### 가능성
- 볼트에 아직 수집되지 않음 — `/ingest`로 소스 추가 후 재질의
- 엔티티는 있지만 inbox에 머물러 있음 — `/classify`로 분류 후 재질의
- 표기 불일치 — 한글·영문·별칭(`aliases:`) 중 다른 표기로 시도

## log
- log.md에 query 이벤트 기록 완료 (참조: 없음, 결과: 모름)
```

### 규칙

- **추측 금지.** "아마 ~일 것이다" 같은 문장 금지. 볼트에 없으면 없다고만 말한다.
- 질의 재구성 제안은 가능한 한 구체적으로. 단순히 "다시 물어보세요"는 불충분.
- 다른 memography 스킬(`/ingest`·`/classify`)로 해결 가능한 경우 안내.
- **"모름"도 log.md에 기록한다** — 어떤 질의가 응답 못 받았는지 추적 가치 (Phase 3).

---

## 공통 규칙

- 답변 언어는 질의 언어를 따른다 (한글 질의 → 한글 답변, 영문 질의 → 영문 답변).
- 파일 경로는 위키링크 `[[...]]` 형식만 사용. 절대 경로나 상대 경로는 사용자 가독성을 해친다.
- 발췌 인용은 `>` 블록쿼트로. 여러 줄이면 줄마다 `>` 붙이거나 한 블록으로.
- **본문 읽기 전용** 원칙 준수 — 페이지 본문은 어떤 것도 수정하지 않는다.
- **log.md만 예외** — Phase 3에 따라 매 호출 1건 append. 이건 hot.md Auto 섹션 운영(touched 집계)에 필수.
- 백링크 추적 표현은 "Phase 2 hop {N}" 또는 "related 추적" 둘 다 통용 (구조화 `related:` 필드 우선).
