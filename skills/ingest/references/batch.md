# 배치 Ingest 플로우

여러 소스를 한 번에 처리하는 모드. 단일 ingest의 7단계 중 **Step 1~3을 통합 확인**으로 압축하고, Step 4~7은 소스별 sequential 자동 실행.

---

## 활성 조건

다음 중 하나면 배치 모드 진입:

- `/ingest` 인자에 소스 2개 이상 (URL·파일 경로 혼합 가능)
- `/ingest --inbox-all` — `01. inbox/` 하위 전체 파일 대상

---

## 준비: 소스 수집

### `--inbox-all`

- `01. inbox/` 재귀 탐색해 **모든 파일** 수집
- 확장자 필터 없음 (지원 안 되는 확장자는 Step 1에서 소스별 에러)
- 제외: hidden 파일(`.`로 시작), 디렉토리

### 리스트 지정

- 인자 순서 그대로 사용
- URL·파일 혼합 가능

수집 결과를 `{sources}` 리스트로 보유.

---

## R1. 전체 요약 (Step 1~2 통합)

### 처리

각 소스에 대해 순차 실행 (Chrome MCP는 싱글 세션이라 sequential 안전):

1. Step 1 (읽기) — URL/파일/Chrome 분기는 기존 v1 워크플로우 준수
2. Step 2 (요약+논점 추출)
3. 결과를 구조화 보관

### 확인 프리젠테이션

```
## R1 소스 요약 ({N}개)

| # | 출처 | 성격 | 한 줄 요약 |
|---|---|---|---|
| 1 | instagram.com/reel/... | SNS 릴스 | {summary1} |
| 2 | 01. inbox/report.pdf | PDF 리포트 | {summary2} |
| 3 | example.com/article | 기사 | {summary3} |
| 4 | ... | ... | ... |

각 소스 핵심 논점:

#1 {title1}:
- {point}
- {point}

#2 {title2}:
- {point}
- {point}
...

---

**전체 방향 OK?** 특정 항목만 조정하려면 "#3 시선 다른 쪽으로" 같이 지시.
```

### 피드백 처리

| 사용자 응답 | 처리 |
|---|---|
| 전체 승인 | R2로 진입 |
| 특정 항목 재작업 | 해당 항목만 Step 2 재실행 후 R1 재제시 |
| 특정 항목 제외 | `{sources}`에서 제거 후 R2로 |

### 에러

- 특정 소스 Step 1 실패 → 해당 항목 제외, 다른 소스 계속. R1 결과에 실패 항목 명시.

---

## R2. 도메인 할당 (Step 3 통합)

### 처리

- `_enterwiki.md`·`_sanawiki.md`·`_sanarchive.md`를 R1 전에 **한 번만 읽어 캐시** (반복 읽기 회피)
- 각 소스의 요약·논점에 classifier.md 규칙 적용

### 확인 프리젠테이션

```
## R2 도메인 할당 ({N}개)

| # | 요약 | 제안 도메인 | confidence | 근거 |
|---|---|---|---|---|
| 1 | {summary1} | sana | high | 음악 취향 |
| 2 | {summary2} | enter | medium | 기획사 IR |
| 3 | {summary3} | archive | low | 디자인 툴 리뷰 (sana 가능성도) |
| 4 | ... | ... | ... | ... |

⚠️ confidence low 항목:
- #3: archive(레퍼런스) vs sana(비주얼) — 어느 쪽?

---

**할당 OK?** override는 "#2 archive로" 같이 지시.
```

### 피드백 처리

- confidence **low는 사용자 판단 필수**. low 해결 전까지 R3 진입 안 함.
- override 받으면 해당 항목 도메인 변경 후 전체 테이블 재제시.

---

## R3. 생성 예정 (Step 4~5 프리뷰)

### 처리

내부적으로 Step 4·5를 **시뮬레이션**만 (실제 파일 생성·이동 없음):

- 제목 LLM 정제
- 경로 결정 (`{domain}/wiki/{category}/{title}.md`)
- 태그 추출
- 원본 처리 방식 결정 (이동 or URL 참조)

### 확인 프리젠테이션

```
## R3 생성 예정 ({N}개 신규 페이지)

| # | 경로 | 제목 | 태그 | 원본 처리 |
|---|---|---|---|---|
| 1 | 03. sanawiki/wiki/음악/{title1}.md | {title1} | [...] | URL (이동 없음) |
| 2 | 02. enterwiki/wiki/기획사례/{title2}.md | {title2} | [...] | inbox → raw/ |
| 3 | 05. sanarchive/wiki/레퍼런스/{title3}.md | {title3} | [...] | URL |
| 4 | ... | ... | ... | ... |

---

**{N}개 전부 생성?** 제외: "#3 제외". 제목 수정: "#1 제목 'XX'로".
```

### 피드백 처리

| 사용자 응답 | 처리 |
|---|---|
| 전체 승인 | 실행 Phase 진입 |
| 개별 제외 | 해당 소스 `{sources}`에서 제거, R3 재제시 |
| 제목 수정 | 해당 항목만 제목 갱신, R3 재제시 |
| 전체 중단 | 아무 파일도 건드리지 않고 종료 |

---

## 실행 Phase (Step 4~7 sequential 자동)

각 소스에 대해 순서대로:

1. **Step 4**: 원본 처리 (파일 이동 또는 URL source 기록)
2. **Step 5**: 소스 페이지 작성 (R3 확정 경로·제목 사용)
3. **Step 6**: 기존 엔티티 링크 삽입 (신규 엔티티 생성 금지, `/promote`에 위임)
4. **Step 7**: `index.md` + `log.md` 갱신

`log.md`는 **소스마다 개별 ingest 이벤트로 기록** (배치 단위 요약 엔트리는 따로 만들지 않음 — 개별 추적성 유지).

### 진행률 업데이트

```
✓ [1/5] {title1} — 03. sanawiki/wiki/음악/
✓ [2/5] {title2} — 02. enterwiki/wiki/기획사례/
⚙ [3/5] 처리 중...
```

### 사용자 확인 없음

R1~R3에서 이미 전체 프리뷰 확인을 받았으므로, 실행 Phase는 전부 자동. 단일 모드와 달리 **소스별 확인 반복 없음**.

---

## 에러 처리 (부분 실패 허용)

**원칙: 하나 실패가 전체를 막지 않는다. 부분 성공 상태 보존.**

| 상황 | 처리 |
|---|---|
| R1에서 소스 읽기 실패 | 해당 소스 제외, 다른 소스 계속 |
| R2에서 confidence low | 사용자 판단 대기, 나머지 진행 안 함 |
| R3에서 경로 충돌 | 해당 소스만 manual mode 안내, 나머지 계속 |
| 실행 Phase의 Step 5 실패 | 그 소스만 에러 로그, 다음 소스 계속 |
| `index.md`·`log.md` 갱신 실패 | 페이지는 생성됨. 수동 갱신 안내 |

---

## 최종 리포트

```
✓ 배치 ingest 완료 (성공 {M} / 전체 {N})

### 성공
- [[path1]] (sana/음악)
- [[path2]] (enter/기획사례)
- [[path3]] (archive/레퍼런스)
...

### 실패 ({N - M}개)
- {source3}: Step 5 경로 충돌 — 수동 확인 필요
- {source5}: Step 1 fetch 실패 — URL 재확인 필요

### 엔티티 후보 (참고)
반복 언급된 이름들. `/promote` 검토 권장:
- {name1}: {page_count}개 페이지
- {name2}: {page_count}개
...

### 관련 페이지 제안 (검토)
- [[page1]], [[page2]]
```

---

## 단일 모드와의 관계

- **단일 소스**: 기존 v1 워크플로우 그대로. 배치 로직 비활성.
- **배치에서 "#3만 단일 모드로" 요청**: 그 소스 배치에서 빼고 단일 `/ingest {source}`로 재호출 안내.

---

## 제약 (v1.1)

- **Sequential 실행**. 진짜 subagent 병렬은 v1.2. 이유: Chrome MCP 싱글 세션, `index.md`·`log.md` 공유 파일 쓰기 충돌 위험.
- **컨텍스트 누적 주의**. 10개 이상 소스는 LLM 컨텍스트 부담 — 대규모 배치는 5~8개 단위로 나눠 실행 권장.
- **confidence low 자동 해결 없음**. R2에서 사용자 판단 필수.
- **`--inbox-all` 필터 없음**. 확장자·패턴 필터는 v1.2.
