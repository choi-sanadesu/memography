# Promote 6단계 상세 워크플로우

각 단계의 입력·처리·출력·에러 처리. CLAUDE.md §8 엔티티 관리 규칙 준수.

---

## Step 1. 중복 체크

### 처리

1. `04. entities/` 하위를 재귀 탐색해 `.md` 파일 basename 목록 수집
2. 입력 이름(정규화 후)과 일치하는 파일 검색
3. 결과 분기:
   - **없음**: Step 2 진행
   - **있음**: 병합 경로 제시

### 병합 경로

```
⚠️ 이미 엔티티 페이지가 있습니다.
- 경로: {existing_path}
- 마지막 갱신: {updated}

새 소스가 추가된 거라면 `참고 소스` 섹션만 갱신할까요? (병합 모드)
아니면 별도 이름으로 새 페이지를 만들까요?
```

### 병합 모드 동작 (사용자 동의 시)

- Step 2(증거 수집)는 실행
- Step 3·4 스킵
- Step 5에서 언급 페이지 백링크는 그대로 추가
- 기존 엔티티 페이지의 `## 참고 소스`에 새 언급 페이지 링크 추가
- 기존 페이지의 `updated:` 필드 오늘 날짜로 갱신
- 기존 본문은 건드리지 않음

---

## Step 2. 증거 수집

### 처리

1. 볼트 전체에서 이름 grep
   - 영문 이름: 대소문자 구분, 가능한 한 word-boundary 적용
   - 한글 이름: 정확 문자열 매칭 (word-boundary 개념 없음)
2. 매칭 위치별 컨텍스트 추출 (전후 2~3줄)
3. 파일별 매칭 횟수 카운트
4. 도메인별 분포 계산

### 제외 대상

- `.trash/`, `.obsidian/`
- `*/raw/**`
- `01. inbox/**`
- 이미 존재하는 엔티티 페이지 (자기 참조 방지)

### 판정 기준 (승격 조건)

v1 기본값:
- 최소 2개 이상의 서로 다른 페이지에서 언급
- 가능하면 2개 이상의 도메인에 걸쳐 있을 것

조건 미충족 시 사용자에게 안내:
```
⚠️ 승격 기준 미충족
- 언급 페이지 수: {N} (기준: 2)
- 도메인 분포: {distribution} (기준: 2+ 권장)

그래도 진행할까요? (단발 언급이면 텍스트로만 남기는 게 적절할 수 있음)
```

### 출력

- 언급 페이지 목록 (경로 + 도메인 + 매칭 횟수)
- 컨텍스트 스니펫 (Step 4의 개요 작성에 활용)
- 공동 언급된 다른 엔티티 후보 (Step 4의 `관련 엔티티` 섹션 힌트)

---

## Step 3. 카테고리 분류

### 처리

Step 2의 컨텍스트를 바탕으로 5개 카테고리 중 판단.

상세 규칙과 edge case는 [category.md](category.md) 참조.

### confidence 판정

| 수준 | 기준 |
|---|---|
| high | 컨텍스트에서 한 카테고리만 뚜렷 (예: "OO 아티스트", "OO 그룹") |
| medium | 두 카테고리 가능성 있으나 맥락상 한쪽 우세 |
| low | 여러 카테고리 해당 가능, 판단 모호 |

### B모드 확인 (low일 때만)

```
## 카테고리 판단

**추천:** {category} (confidence: low)
- 근거: {한 줄}

**대안 후보:**
- {cat2}: {근거}
- {cat3}: {근거}

어느 카테고리로 진행할까요?
```

high·medium은 사용자 확인 없이 Step 4로 진행.

## Step 4. 엔티티 페이지 구성

### 경로

```
04. entities/{category}/{name}.md
```

- `{category}`: 한글 카테고리명 (`인물`·`브랜드`·`작품`·`개념`·`플랫폼`)
- `{name}`: 정규화된 이름 (SKILL.md 이름 정규화 규칙)

카테고리 폴더가 미존재하면 생성 (`mkdir -p`).

### 템플릿 (CLAUDE.md §4.1 준수)

```yaml
---
title: {name}
type: entity
domain: entity
tags: [{category 기반 + 컨텍스트 키워드 2~3개}]
updated: {YYYY-MM-DD}
---

# {name}

## 개요

{2~3문장. Step 2의 컨텍스트를 합성. 중립적 사실 위주.}

## 핵심 정보

- 분류: {category 구체화 — 예: "인물/솔로 아티스트", "브랜드/엔터사"}
- 관련: {컨텍스트에서 뚜렷하게 등장한 소속·관계. 예: "ADOR 소속", "하이브 자회사"}

## 상세

{비워둠 또는 placeholder. 사용자가 시간을 두고 채움.}

## 관련 엔티티

{Step 2에서 공동 언급된 다른 엔티티 페이지가 있으면 링크로. 없으면 비움.}
- [[...]]

## 참고 소스

{Step 2의 언급 페이지 전부를 경로 기반 링크로.}
- [[{path1}|{title1}]]
- [[{path2}|{title2}]]
```

### 내용 작성 원칙

- **중립적 사실 위주** (CLAUDE.md §4.2 원칙). 각 도메인의 시선(업계/개인 취향/작업 맥락)은 도메인 페이지에서 다룸.
- **과장 금지**. 컨텍스트에 없는 정보를 추론하지 않음.
- **상세 섹션은 비워두기**. 사용자가 점진적으로 채우는 걸 전제. placeholder 금지 (빈 상태가 명확).

### B모드 확인

**신규 페이지 생성이므로 확인 필수:**

```
## 생성 예정: [[04. entities/{category}/{name}]]

---
title: {name}
type: entity
domain: entity
tags: [{tags}]
updated: {date}
---

# {name}

## 개요
{preview}

## 핵심 정보
- 분류: {...}
- 관련: {...}

## 참고 소스
- [[{path1}]] ({domain1})
- [[{path2}]] ({domain2})
...

생성해도 될까요?
```

confirm 후 `Write`.

## Step 5. 백링크 업데이트

### 대상

Step 2의 언급 페이지 전부. 각각에 대해:

1. 현재 페이지의 `## 관련` 섹션 확인
   - 없으면 생성
   - 있으면 중복 체크
2. 새 엔티티 링크 추가

상세 로직은 [backlink.md](backlink.md) 참조.

### 자동 적용

- 이 단계는 **자동 실행** (사용자 확인 없이 Edit)
- CLAUDE.md §5.2 규칙: "기존 페이지 수정은 바로 진행 후 변경 요약 보고"
- 하지만 **수정 후 영향받은 페이지 목록을 Step 6 완료 보고에 포함**

### 예외

사용자가 "백링크는 수동으로 하겠다"고 명시하면 이 단계 건너뛰고 리포트만.

## Step 6. index.md + log.md 갱신

### index.md

`## 04. entities` → `### {category}` 섹션에 한 줄 추가:

```markdown
- [[04. entities/{category}/{name}|{name}]] — {한 줄 요약 (개요 첫 문장 압축)}
```

카테고리 섹션이 `_아직 없음_`으로 비어 있으면 그 항목을 제거하고 새 링크로 대체.

### log.md

append-only, 앞쪽 `---` 뒤에:

```markdown
## [{YYYY-MM-DD}] promote | {name}

- 도메인: entity
- 추가된 페이지: [[04. entities/{category}/{name}]]
- 갱신된 페이지: [[index]] + {백링크 추가된 페이지 목록}
- 핵심 인사이트: {왜 승격했는지, 어떤 맥락에서 반복 언급됐는지 2~3문장}
```

### 병합 모드의 경우

- `추가된 페이지: 없음 (기존 엔티티 갱신)`
- `갱신된 페이지: [[existing_entity]], [[index]], ...`
- `핵심 인사이트: 기존 엔티티에 참고 소스 N개 병합`

## 완료 출력

```
✓ promote 완료

- 엔티티: {name} ({category})
- 신규 페이지: [[04. entities/{category}/{name}]]
  또는: 병합 완료 — [[existing_path]]에 참고 소스 {N}개 추가
- 갱신된 페이지: [[index]], [[log]]
- 백링크 추가: {N}개 페이지
  - [[{page1}]]
  - [[{page2}]]
  ...
- 공동 언급 엔티티 힌트 (참고): {other_name1}, {other_name2} → 각각 /promote 검토 가능
```

## 에러 복구

| 실패 단계 | 상태 | 처리 |
|---|---|---|
| Step 1~3 | 변경 없음 | 에러 출력만 |
| Step 4 | 쓰기 실패 | 엔티티 페이지 없음. 재실행하면 중복 체크가 다시 돎 |
| Step 5 | 일부 페이지 백링크 실패 | 이미 성공한 페이지는 그대로. 실패 페이지 목록 보고 |
| Step 6 | index/log 실패 | 엔티티 페이지·백링크는 반영됨. index/log 수동 갱신 안내 |

원칙: 파괴적 롤백 금지. 부분 성공 상태에서 다음 재실행이 이어서 처리할 수 있게.
